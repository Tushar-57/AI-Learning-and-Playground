

# OPEN AI's - PARAMETER GOLF CHALLANGE - Deconstructing LLM Architecture 🧠


These are my deep-dive revision notes on building a transformer from the ground up, specifically focusing on the mathematical mechanics and data flow. The motivation, challenge and context here is the "Parameter Golf" constraint: training a model under a strict 16MB limit, which forces a brutal understanding of where every byte goes.

## 1. The Core Budget: Parameters vs. Bytes
Before touching architecture, the storage math dictates everything. A parameter (a weight or a bias) is essentially an "importance number" that the model tweaks during training. 

* **Float32 (Standard Training):** 1 parameter = 4 bytes. A 1.28 million parameter layer costs 5.12 MB. We *must* use floats during training because the backpropagation algorithm relies on microscopic adjustments (like a scalpel) to find the correct vector space.
* **INT8 (Quantization):** 1 parameter = 1 byte. That same 1.28M parameter layer becomes 1.28 MB.
* **Post-Training Quantization (PTQ):** We train in floats, then map the float range (e.g., -2.5 to +3.8) to an INT8 range (-128 to 127). We lose decimal resolution, but this is acceptable because the model cares more about the *direction/topology* of the vector in space than its exact coordinates. 

## 2. The Embedding Layer: The "Lookup Table"
The embedding layer maps discrete token IDs to dense vectors of floating-point numbers. It is an $O(1)$ matrix fetch operation, grabbing a specific row (e.g., Row 42 for the word "Apple") from a massive 2D array.

* **The Cost Formula:** `Vocabulary Size (V) × Embedding Dimension (d)`. For a 1024 vocab and 512 dimension, this matrix holds 524,288 parameters.
* **Weight Tying:** To save space, the input embedding matrix and the output "unembedding" matrix share the exact same weights, immediately recycling roughly 0.5 MB of space in our baseline.
* **Dimensionality ($d$): 512 vs 128:** Why not use a smaller dimension to save space? "Crowding". In a 128-D space, complex datasets force unrelated words to map to the same coordinates, causing "interference" (like flattening a 3D city map into 2D where bridges and roads collide). High dimension (breadth) provides the necessary "corners" for distinct concepts to live separately.

## 3. The Attention Mechanism: The "Context" Engine
Attention acts horizontally across tokens to determine relationships. It takes the static dictionary vector from the embedding layer and makes it dynamic.

### The Q, K, V Handshake
Every token vector is multiplied by three separate weight matrices (each $512 \times 512$) to create three new vectors:
1. **Query (Q):** "What am I looking for?".
2. **Key (K):** "What do I contain?".
3. **Value (V):** "What is my actual underlying data/message?".

### The Matrix Multiplications & Data Flow
Assume a sequence of 10 tokens, each 512-D.
1. **The Scores:** $Q \cdot K$. We calculate the dot product of every word against every other word, creating a $10 \times 10$ raw score matrix. 
2. **Softmax:** We exponentiate these raw scores and divide by the sum so each row perfectly equals 1.0 (100%). This turns raw affinity into exact percentage weights.
3. **The Value Projection:** The input ($10 \times 512$) is multiplied by the Value weights ($512 \times 512$) to create the Value Matrix ($10 \times 512$). If we aggressively compress the weights to $512 \times 64$, the Value Matrix becomes a $10 \times 64$ Data Transfer Object (DTO)—losing breadth but forcing the token to send only its most critical features.
4. **The Weighted Sum:** We multiply the $10 \times 10$ Attention Scores by the $10 \times 512$ Value Matrix. Row 1 of the scores weights the 10 Value rows, summing them into a brand new 512-D vector. The static token has now "absorbed" its context.

* **The Self-Attention Paradox:** Why does a $10 \times 10$ matrix calculate a word against itself? If "cat" attends 100% to "sat" and 0% to itself, the weighted sum completely replaces the "cat" vector with the "sat" vector. The subject becomes an action. Self-attention anchors the token's core identity.

## 4. Feed-Forward Networks (FFN): The "Database"
If Attention is the navigation engine (routing "Who" to "Prime Minister"), the FFN is the actual database where hard facts ("Justin Trudeau") are stored in the weights. It works independently on each token, moving information vertically up the abstraction ladder.

### The Expansion and Filtering Math
To do complex "IF-THEN" logic, the FFN temporarily expands the vector's dimensions to create a massive scratchpad. 
* **Up-Projection:** Multiply the 512-D vector by a $512 \times 2048$ weight matrix. (Cost: $512 \times 2048 = 1,048,576$ parameters).
* **Activation (GELU):** A non-linear filter is applied to the 2048-D space. It pushes negative or weak numbers to zero, killing off noise and keeping strong semantic signals. 
* **Down-Projection:** Multiply the filtered vector by a $2048 \times 512$ matrix to compress it back to the main highway size. (Cost: $2048 \times 512 = 1,048,576$ parameters).

* **Why not keep it at 2048-D permanently?** A permanent 2048-D highway would cause a "Memory Wall," exploding the parameter count by 16x ($d^2$). Compressing back to 512-D forces the model through an "information bottleneck," forcing it to learn to summarize the most important features rather than getting lazy. Under strict memory limits, shrinking the FFN expansion multiplier (e.g., from 4x to 2x) saves massive space at the cost of the model's "subject knowledge," whereas dropping attention heads hurts its "social skills" to read context.

## 5. Residual Connections: The "Highway"
With deep networks (e.g., 9+ layers of complex Attention and FFN math), the original semantic meaning of a word can get completely distorted by "math noise". 
* **The Formula:** $Output = x + FFN(x)$.
By physically taking the original input vector ($x$) and adding it back to the processed output, we create a skip connection highway. This guarantees that the core identity of the token survives all the way to the final layer.

## tl;dr

## 1. Embeddings: Dimensionality vs. Precision
Initially, I was confused about how we choose embedding sizes (like 128D vs 512D) and how words actually map in that space. My engineering brain thought that if we had highly similar words, we should use higher precision numbers (like Float32) in a smaller space (512D) so the "knobs" could be fine-tuned accurately. 

**What I actually learned:**
* **Topology > Precision**: AI cares way more about the *direction* of the vector than the 7th decimal place. A 1024-dimensional space using low-precision integers (INT8) is actually much better than 512D using Float32. More dimensions mean more "corners" in the vector space, preventing unrelated words from crowding together or interfering with each other.
* **Post-Training Quantization (PTQ)**: You can't *train* a model in INT8 because the adjustments are too microscopic—it's like trying to do surgery with a sledgehammer. You train in Float32 (the scalpel) and then compress to INT8 later. The minor loss of resolution is fine, but a loss of dimension would be a disaster because the spatial space would become too crowded.

## 2. The Attention Mechanism & The "Identity" Problem
I struggled with the "Weighted Sum" step and why the Attention score matrix calculates relationships in an `N x N` grid instead of `N x (N-1)`. Why would we waste compute calculating a word's weight against itself?

**What I actually learned:**
* **The Self-Attention Paradox**: If a word like "cat" only looked at its surrounding context (like "sat") and ignored itself, its vector would be 100% replaced by the context. It would basically forget it's an animal and become an action. Self-attention acts as the anchor; it ensures the core identity of the word stays in the mix.
* **The Mixing Bowl**: Attention takes static, dictionary-like embedding vectors and physically drags them toward a specific contextual meaning using a weighted average of the surrounding words.
* **Value Projections**: When passing data along, the model compresses the 512D vector into a smaller dimension (like 64D). It acts like a DTO (Data Transfer Object) in software engineering—it loses breadth, but forces the model to only send the most critical features to the next layer without the noise.

## 3. Feed-Forward Networks (FFN): Navigation vs. Database
I was initially choppy on the difference between what Attention does versus what the FFN does. I incorrectly assumed Attention handled broad relationships and FFN figured out detailed grammatical relationships.

**What I actually learned:**
* **The Office Analogy**: Attention is like a meeting where words talk to each other horizontally across the sentence to get context. FFN is "deep work" at a private desk where the vector moves vertically up the abstraction ladder, looking internally.
* **Where Facts Live**: Attention is just *Navigation* (routing the word "Who" to the phrase "Prime Minister"). The FFN is the actual *Database* that stores the hard fact that "Prime Minister of Canada" = "Justin Trudeau".
* **The Temporary Expansion Trick**: Inside the FFN, the vector is temporarily expanded (e.g., from 512D to 2048D) to create a massive scratchpad for complex IF-THEN logic, passed through a GELU filter to kill off weak signals, and then compressed back down. I wondered why we don't just keep the highway at 2048D permanently. It turns out that would cause a "Memory Wall" (parameter explosion) and prevent the model from learning to summarize important features.

## Final Thought on Architecture Constraints
When you are limited to 16MB, every byte is a liability. A standard FFN expansion is the biggest space hog in a Transformer. Dropping the FFN multiplier saves massive space but costs the model its "subject knowledge," whereas dropping Attention heads hurts the model's "social skills" to read context. Understanding these trade-offs mathematically is the fundamental difference between wrapping an API and architecting a model.
